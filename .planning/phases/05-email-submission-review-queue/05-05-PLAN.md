Phase 4 does NOT persist the tailored JSON — only the rendered DOCX at `TailoringRecord.tailored_resume_path`.
```python
"""Round-trip a tailored DOCX through user edits (no LLM, no validator)."""
from __future__ import annotations

from pathlib import Path
from typing import Any

from docx import Document


def extract_sections_from_docx(docx_path: Path | str) -> dict[str, Any]:
    """Walk headings in a DOCX and return the `{sections: [...]}` shape
    that `build_tailored_docx` accepts as input.

    Simple heuristic: every paragraph whose style name starts with 'Heading'
    is a section boundary. Content is the list of non-heading paragraphs
    until the next heading.

    This does NOT attempt to reconstruct the Experience->subsections shape
    that the LLM originally emitted — the user edits at the section level;
    subsections flatten into `content` lines. That matches the locked
    decision ('edit bullets in the browser') and keeps the round-trip
    stable because `build_tailored_docx` treats a flat `content` list
    as bullets inside the section regardless.
    """
    doc = Document(str(docx_path))
    sections: list[dict[str, Any]] = []
    current: dict[str, Any] | None = None
    for para in doc.paragraphs:
        style = (para.style.name or "") if para.style is not None else ""
        if style.lower().startswith("heading"):
            if current is not None:
                sections.append(current)
            current = {"heading": para.text.strip(), "content": []}
        else:
            if current is None:
                # pre-heading paragraphs (name/contact block) — bucket as a
                # pseudo-section the user can optionally edit. Use an empty
                # heading so build_tailored_docx ignores it on write but the
                # textarea still appears in the UI.
                current = {"heading": "", "content": []}
            text = para.text.strip()
            if text:
                current["content"].append(text)
    if current is not None:
        sections.append(current)
    return {"sections": sections}


def apply_user_edits(
    *,
    base_resume_path: Path,
    edited_sections: dict[str, Any],
    output_path: Path,
) -> Path:
    """Regenerate a tailored DOCX from user-edited sections.

    Pipes straight through `build_tailored_docx` which uses the only
    safe mutator in this codebase (`replace_paragraph_text_preserving_format`).
    NEVER call `paragraph.text = ...` directly (Phase 4 pitfall 1).
    """
    from app.tailoring.docx_writer import build_tailored_docx
    return build_tailored_docx(
        base_resume_path=base_resume_path,
        tailored_sections=edited_sections,
        output_path=output_path,
    )


__all__ = ["extract_sections_from_docx", "apply_user_edits"]
```

- `async def approve_one(session, job_id) -> None`:
  - Loads Job; calls `assert_valid_transition(job.status, 'approved')` from `app.review.states`; writes `job.status='approved'`; commits.

- `async def approve_batch(session, job_ids: list[int]) -> int`:
  - Loops approve_one inside a single transaction; returns count of approved. Any ValueError for illegal transitions raises and rolls back (all-or-nothing).

- `async def reject_job(session, job_id, *, mode: str) -> None`:
  - `mode` must be `'skip'` or `'retailor'`.
  - `skip`: flip to `skipped` via transition guard.
  - `retailor`: flip to `retailoring`. A follow-up scheduler run will notice `retailoring` and pick it up through the existing Phase 4 engine. **Add a matching branch in `app/tailoring/pipeline.py::run_tailoring`** that selects jobs with `status IN ('matched', 'retailoring')` — verify by reading the existing SELECT in that file. If the existing query only selects `matched`, extend it to include `retailoring` in the `status IN (...)` clause. Keep the single-line patch minimal.

- `async def save_user_edits(session, job_id, edited_sections: dict) -> TailoringRecord`:
  - Uses `app.tailoring.service.get_next_version(job_id)` + `resume_artifact_path(job_id, version)` to pick the new output path.
  - Calls `apply_user_edits(base_resume_path=existing_base, edited_sections=edited_sections, output_path=new_path)`.
  - Inserts a new TailoringRecord with `intensity='manual_edit'`, `status='completed'`, `validation_passed=None`, `retry_count=0`, `error_message='user_edit'`, paths populated, cost fields zero. Do NOT debit budget.
  - Commits and returns the new record.

Every status-writing function MUST go through `assert_valid_transition`. No raw `job.status = ...` writes outside that guard.
