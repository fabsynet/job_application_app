**Step 1 — verify retry_count semantics (research open question #3).**

Before writing `holdout.py`, READ `app/tailoring/engine.py` lines 379 + 396-400 + 513 + 526 + 567 + 587. The `retry` loop index starts at 0 and `retry_count = retry + 1` is written on both success (line 567/587) and validation-failure (line 513/526). Therefore:

- **`retry_count == 1` means first-try success** (retry loop index 0 succeeded and we wrote `retry + 1 = 1`).
- **`retry_count == 0`** only occurs on catastrophic engine failure (line 379 path — no retries ran).
- **`retry_count >= 2`** means validator rejected at least once.

The holdout therefore uses `record.retry_count <= 1 AND record.validation_passed is True`. Document this in the module docstring so the verifier can cross-check against the engine.

**Step 2 — write `app/submission/holdout.py`:**

```python
"""Phase 5 low-confidence holdout decision.

Locked: auto-submit only when BOTH conditions hold:
  1. Validator passed on first try (TailoringRecord.validation_passed is True
     AND TailoringRecord.retry_count <= 1 — see engine.py line 567 for the
     retry+1 semantics; retry_count==1 is first-try success).
  2. Keyword coverage >= user match threshold + holdout margin.
"""
from __future__ import annotations

from dataclasses import dataclass

from app.discovery.models import Job
from app.tailoring.docx_writer import compute_keyword_coverage
from app.tailoring.models import TailoringRecord


@dataclass(frozen=True)
class HoldoutDecision:
    auto_eligible: bool
    reason: str   # stable code for logs + queue badge
    coverage_pct: int
    required_pct: int


def should_auto_submit(
    *,
    record: TailoringRecord,
    job: Job,
    tailored_text: str,
    user_threshold: int,
    holdout_margin_pct: int,
) -> HoldoutDecision:
    if record.validation_passed is not True:
        return HoldoutDecision(False, "validation_not_passed", 0, 0)
    # retry_count==1 is first-try success per engine.py line 567.
    if (record.retry_count or 0) > 1:
        return HoldoutDecision(False, "validation_needed_retries", 0, 0)
    coverage = int(compute_keyword_coverage(tailored_text, job.description) * 100)
    required = max(0, min(100, user_threshold + holdout_margin_pct))
    if coverage < required:
        return HoldoutDecision(False, "coverage_below_holdout", coverage, required)
    return HoldoutDecision(True, "auto_eligible", coverage, required)


__all__ = ["should_auto_submit", "HoldoutDecision"]
```

Algorithm (copy and tailor from research Pattern 1 + 2):

1. **Pause check.** Load `Settings` row. If `submissions_paused is True`, log `submission_paused` and return `_empty_counts(paused=True)` immediately. Discovery + tailoring have already run; the queue will just fill up.

2. **Auto-mode holdout + review-mode branch.** Load all jobs with `status == 'tailored'` (the review queue holding pen). For each:
   - Look up the latest `TailoringRecord` via `app/tailoring/service.py::get_latest_tailoring` (use the existing helper). Skip if no completed record.
   - Extract `tailored_text` by reading paragraphs from `record.tailored_resume_path` via `Document(path).paragraphs` (same primitive as `extract_cover_letter_plaintext` in 05-02 — plant a helper in `app/submission/builder.py` called `extract_docx_plaintext(path) -> str` if not already present).
   - If `Settings.auto_mode` is True, call `should_auto_submit(...)` with `Settings.match_threshold` and `Settings.auto_holdout_margin_pct`. If `auto_eligible`, flip `tailored -> approved` via `flip_job_status`. Otherwise leave it in `tailored` (it'll show up in the review queue) — increment `held_out` counter and log the reason.
   - If `Settings.auto_mode` is False, leave `tailored` alone — the user approves manually via 05-05's UI.

3. **Drain approved queue.** Call `list_approved_jobs(session)`. For each `(job, record)`:
   a. `await killswitch_check()` between jobs.
   b. **Quiet hours check.** Read `Settings.quiet_hours_start/end`. Compute "current hour in `Settings.timezone` via `zoneinfo.ZoneInfo`". If current hour is within quiet hours (handle wrap-around for end < start), log `quiet_hours_skip` and break out of the drain loop — do NOT mark failed, just leave `approved` for the next run. Document this as "Phase 5 is first enforcement site for quiet hours" in the module docstring (research surprise finding).
   c. **Daily cap precheck.** Call `rate_limiter.await_precheck(session)`. If raises `RateLimitExceeded`, set `counts["rate_limited"] = True`, log `daily_cap_halt`, and break out of the drain loop. Remaining approved jobs stay in `approved` for next run — matches SC-2 locked decision.
   d. **Resolve recipient.** `resolve_recipient_email(job.description)`. If None: `flip_job_status(job.id, 'needs_info')`, emit `send_failure_notification(session, signature=..., stage='submission', ...)` with a dedicated `error_class='NoRecipientEmail'`, increment `needs_info`, continue.
   e. **Select strategy.** `strategy = select_strategy(job, job.description, registry)`. If None: same `needs_info` branch.
   f. **Build context.**
      - Load SMTP creds via `load_smtp_creds(session)` once at the top of the drain loop (cache it; do NOT reload per job).
      - `subject = build_subject(role=job.title, company=job.company)`
      - `body_text = extract_cover_letter_plaintext(record.cover_letter_path)`
      - `attachment_filename = build_attachment_filename(full_name=profile.full_name, company=job.company)` — load Profile once, same cache pattern.
      - `recipient_email = resolve_recipient_email(job.description)`
   g. **Insert pending Submission row.** `insert_pending(...)`. This runs in its own short-lived session so a subsequent crash doesn't lose the audit trail.
   h. **Call strategy.submit(ctx).** On success → `mark_sent(submission_id)`, `flip_job_status(job.id, 'submitted')`, `rate_limiter.record_submission(session)`, `send_success_notification(session, job, record, submission_id)`, `clear_suppressions_for_stage(session, 'submission')` (clear any prior failure-suppressions because the next success signals recovery), increment `submitted`.
      - If `mark_sent` raises `IdempotentDuplicate`: that means another path already marked this job `sent` — log, increment `submitted`, do NOT notify twice, continue.
   i. On failure outcome → build signature via `build_signature(error_class, 'submission', error_message)` (function lives in 05-07's notifications module), `mark_failed(submission_id, ...)`, `flip_job_status(job.id, 'failed')`, `await send_failure_notification(session, signature, stage='submission', error_class, error_message, job)`, increment `failed`.
   j. **Delay.** `await asyncio.sleep(rate_limiter.random_action_delay())` between jobs (respects SAFE-02).
