## User Constraints (from CONTEXT.md)

### Locked Decisions

**Secrets bootstrap:**
- `FERNET_KEY` delivered via env var at `docker compose up`. Not generated by the app, not mounted as a file. User stores the key outside the container.
- API keys / SMTP credentials entered via the Settings page web UI form. Fields are encrypted with the master key and stored in the SQLite `secrets` table on save. No env-var seeding of API keys — UI is the only entry path.
- No key rotation tool in v1. If master key is lost/changed, stored secrets become unreadable; user re-enters credentials via UI. Document this.
- Zero-tolerance leak policy: global log scrubber redacts any value matching registered secret patterns (API keys, passwords, Fernet key, SMTP creds) before reaching stdout / log files / error responses. Ship with a test asserting known-secret patterns never appear in log output.

**Control panel UX (Phase 1 landing surface):**
- Home page is a **live status dashboard** showing: scheduler state (running / paused / killed), next scheduled run time, last run timestamp + duration + outcome, counts from last run (discovered / matched / tailored / submitted / failed — all zero in Phase 1), prominent **Kill switch** and **Dry run** toggles.
- Kill-switch = hard stop: cancels in-flight run immediately; any partially-processed records marked `failed` with reason `killed`. Scheduler pauses future runs until switch flipped back.
- Dry-run = real reads, no writes: scrapers fetch, LLM runs, DB observability rows still written, but **no outbound submissions** and **no outbound notification emails**. Every record created during the run is stamped with a `dry_run` flag.
- Run history: DB keeps every run row forever (hourly runs ≈ 720/month, trivial for SQLite). UI shows 50 most recent with "show more" affordance. No pruning.

**Rate-limit envelope defaults:**
- Daily application cap: 20/day global, user-configurable in Settings.
- Per-action delay: randomized 30–120s. Phase 1 stores range and enforces envelope; Phase 6 first consumes the per-action delay.
- Single global cap across all sources, no per-source override in v1.
- Counter resets at local midnight in container TZ (driven by `TZ` env var), NOT a rolling 24h window.
- Rate-limit envelope enforced by the scheduler **before any downstream stage exists** so later phases inherit throttling for free.

**Docker & volumes:**
- `docker compose up` with `compose.yml` + `.env` file. `.env.example` documents required vars (`FERNET_KEY`, `TZ`, `BIND_ADDRESS`).
- Single `./data` host volume mounted into container — SQLite db, logs, uploaded base DOCX, Playwright storageState, tailored artifacts. One directory to back up, one to wipe for reset.
- LAN-bound by default (`0.0.0.0`); configurable via `BIND_ADDRESS` env var.
- First-run experience: setup wizard. Fresh `./data` → UI routes to wizard: (1) upload base DOCX resume, (2) enter Claude API key + SMTP creds, (3) set target keywords. Wizard writes complete-flag to DB. Wizard is skippable, not a gate.

- Exact `runs` / `secrets` / `settings` table schemas (as long as log-scrubber test and 50-row UI work).
- UI component breakdown within Phase 1 (HTMX + Jinja are locked; page structure is not).

## Summary

Phase 1 is the bootable safety frame: a FastAPI + APScheduler monolith that runs hourly heartbeats, survives restarts with Fernet-encrypted secrets, enforces kill-switch and dry-run toggles at the scheduler level, and guarantees zero PII leakage via a global log scrubber. No pipeline stages exist yet — but every contract later phases consume is locked here so Phases 2–7 "plug into" rather than "bolt on" the safety envelope.

The single most important architectural decision is that **kill-switch, dry-run, and rate-limit checks are gates the scheduler enforces before the pipeline function is even called** (and re-checked at stage boundaries once stages exist). This means later phases inherit throttling automatically rather than having to remember to check flags.

**Primary recommendation:** APScheduler **3.11.x** (stable), `AsyncIOScheduler` started in a FastAPI `lifespan` context manager, in-process singleton run-lock via an `asyncio.Lock` + DB `runs.status='running'` sentinel row, kill-switch wired through an `asyncio.Event` + `asyncio.Task.cancel()`, dry-run flag stamped on every `Run` row from the settings table, Fernet master key loaded from `FERNET_KEY` env var with fail-fast validation at `lifespan` entry, global log scrubber implemented as a `logging.Filter` attached to the root logger plus a structlog processor mirror, and a `compose.yml` with a single service binding `./data:/data`.

## APScheduler 3.x vs 4.x: Decision

**Decision: APScheduler 3.11.x.**

**Evidence:**
- Latest stable on PyPI as of April 2026 is 3.11.x. 4.0 has had alphas since at least April 2025 (`4.0.0a6`) with no final release. ([APScheduler PyPI](https://pypi.org/project/APScheduler/))
- Agronholm's own migration doc flags v4 as pre-release, backwards-incompatible without migration path.
- `AsyncIOScheduler` in 3.11 runs jobs in the same asyncio loop as FastAPI — exactly what we need. The 4.x `AsyncScheduler` (AnyIO-based, async-context-manager API) is more elegant but not worth betting on alpha.
- Community consensus (Sentry, Medium writeups, grillazz/fastapi-apscheduler) all use 3.x `AsyncIOScheduler` + FastAPI lifespan. ([Sentry — Schedule tasks with FastAPI](https://sentry.io/answers/schedule-tasks-with-fastapi/))

**Upgrade path:** 4.x becomes viable when Agronholm tags a stable 4.0.0 (watch `apscheduler/releases`). Our pipeline code touches the scheduler via a thin `SchedulerService` wrapper so a 3→4 swap is contained.

### Pattern 1: FastAPI lifespan hosts the scheduler singleton

**What:** The `AsyncIOScheduler` and `SchedulerService` are created in `main.py`, started inside `lifespan`, and attached to `app.state` so any request handler can observe them.

**Why:** Same event loop as FastAPI handlers, no extra thread, no broker. Shutdown is deterministic.

```python
# app/main.py
from contextlib import asynccontextmanager
from fastapi import FastAPI
from apscheduler.schedulers.asyncio import AsyncIOScheduler
from apscheduler.triggers.cron import CronTrigger
from apscheduler.jobstores.sqlalchemy import SQLAlchemyJobStore

from app.config import settings
from app.db.base import init_db, engine
from app.logging_setup import configure_logging
from app.security.fernet import FernetVault
from app.scheduler.service import SchedulerService
from app.scheduler.heartbeat import heartbeat_job
from app.scheduler.killswitch import KillSwitch
from app.scheduler.rate_limit import RateLimiter

@asynccontextmanager
async def lifespan(app: FastAPI):
    # 1. Configure logging BEFORE anything else can log
    configure_logging(settings.log_level, settings.data_dir / "logs")

    # 2. Fail fast on invalid Fernet key
    app.state.vault = FernetVault.from_env(settings.fernet_key)  # raises on invalid key

    # 3. DB
    await init_db()

    # 4. Load stored secrets and register them with the log scrubber
    await app.state.vault.register_all_secrets_with_scrubber()

    # 5. Scheduler
    jobstore_url = f"sqlite+pysqlite:///{settings.data_dir}/app.db"
    scheduler = AsyncIOScheduler(
        jobstores={"default": SQLAlchemyJobStore(url=jobstore_url)},
        timezone=settings.tz,
    )
    app.state.killswitch = KillSwitch()
    app.state.rate_limiter = RateLimiter.from_settings_row()
    app.state.scheduler = SchedulerService(
        scheduler=scheduler,
        killswitch=app.state.killswitch,
        rate_limiter=app.state.rate_limiter,
    )

    scheduler.add_job(
        heartbeat_job,
        trigger=CronTrigger(minute=0, timezone=settings.tz),
        id="hourly_heartbeat",
        replace_existing=True,
        max_instances=1,           # APScheduler-level overlap guard
        coalesce=True,             # missed runs collapse to one
        misfire_grace_time=300,
    )
    scheduler.start()

    try:
        yield
    finally:
        scheduler.shutdown(wait=True)
        await engine.dispose()

app = FastAPI(lifespan=lifespan)
```

Sources: APScheduler docs (AsyncIOScheduler), [Sentry — Schedule tasks with FastAPI](https://sentry.io/answers/schedule-tasks-with-fastapi/), [Running Scheduled Jobs in FastAPI](https://www.nashruddinamin.com/blog/running-scheduled-jobs-in-fastapi).

### Pattern 2: Singleton run-lock (in-process)

**What:** Only one pipeline run may execute at a time. Enforced two ways defensively:
1. `max_instances=1` + `coalesce=True` on the APScheduler job definition (cheap, correct for most cases).
2. `asyncio.Lock` inside `SchedulerService.run_pipeline()` — if a manual force_run overlaps with the cron tick, the second waits (or fast-fails if caller passes `wait=False`).
3. DB sentinel: `Run.status == 'running'` row visible to the dashboard; leftover-on-crash rows cleaned up on startup (`mark_orphans_failed`).

**Why not flock/pidfile:** Uvicorn runs with `--workers 1`, so a single Python process owns the scheduler. File locks are redundant and add Windows/Linux differences. The DB sentinel row is what the dashboard and operator recovery care about.

```python
# app/scheduler/service.py
import asyncio
from contextlib import asynccontextmanager

class SchedulerService:
    def __init__(self, scheduler, killswitch, rate_limiter):
        self._scheduler = scheduler
        self._killswitch = killswitch
        self._rate_limiter = rate_limiter
        self._lock = asyncio.Lock()
        self._current_task: asyncio.Task | None = None

    async def run_pipeline(self, *, triggered_by: str):
        if self._killswitch.is_engaged():
            await log_run_skipped(reason="kill_switch_engaged")
            return
        async with self._lock:
            await self._rate_limiter.await_precheck()   # raises RateLimitExceeded if daily cap hit
            run = await create_run(
                dry_run=await get_setting("dry_run"),
                triggered_by=triggered_by,
            )
            task = asyncio.create_task(_execute_run(run, self._killswitch))
            self._current_task = task
            try:
                await task
            except asyncio.CancelledError:
                await mark_run_killed(run.id)
                raise
            finally:
                self._current_task = None

    def cancel_current_run(self) -> bool:
        if self._current_task and not self._current_task.done():
            self._current_task.cancel()
            return True
        return False
```

### Pattern 3: Kill-switch = engage flag + task cancellation

**What:** `KillSwitch` wraps a shared `asyncio.Event` plus a boolean in the `settings` table. `engage()` sets both, pauses the APScheduler job, and calls `SchedulerService.cancel_current_run()`. `release()` clears the event, unpauses the job, and writes back to settings.

**Why `asyncio.Event` + `asyncio.CancelledError`:** In an async pipeline, `task.cancel()` propagates `CancelledError` through every `await`, which is exactly the semantics we want — in-flight HTTP calls, DB ops, and sleeps all abort. Later stages can also poll `killswitch.is_engaged()` at stage boundaries for a graceful check.

```python
# app/scheduler/killswitch.py
import asyncio

class KillSwitch:
    def __init__(self):
        self._event = asyncio.Event()

    def is_engaged(self) -> bool:
        return self._event.is_set()

    async def engage(self, scheduler_service, settings_service):
        self._event.set()
        await settings_service.set("kill_switch", True)
        scheduler_service.pause_scheduler()          # stops future APScheduler ticks
        scheduler_service.cancel_current_run()       # aborts in-flight task

    async def release(self, scheduler_service, settings_service):
        self._event.clear()
        await settings_service.set("kill_switch", False)
        scheduler_service.resume_scheduler()

    async def raise_if_engaged(self):
        """Stage-boundary checkpoint for graceful abort."""
        if self._event.is_set():
            raise asyncio.CancelledError("kill_switch_engaged")
```

Post-cancel cleanup: `SchedulerService.run_pipeline()`'s `except asyncio.CancelledError` block calls `mark_run_killed(run.id)`, which sets `Run.status='failed'`, `Run.failure_reason='killed'`, and updates any partial records created during the run (none in Phase 1, Phases 3+ inherit).

### Pattern 4: Dry-run propagation via `RunContext`

**What:** The `dry_run` flag lives in the `settings` table, is read at run-start, written onto the `Run` row, and passed downstream via a `RunContext` dataclass. Phase 1 locks the contract: every stage signature is `async def stage(ctx: RunContext, ...)` and every record created by a stage takes `dry_run=ctx.dry_run`.

**Why not a ContextVar:** Explicit is better than implicit. ContextVars leak across tasks in subtle ways; a dataclass argument is obvious in code review and trivial to test.

```python
# app/runs/context.py
from dataclasses import dataclass
from datetime import datetime

@dataclass(frozen=True)
class RunContext:
    run_id: int
    started_at: datetime
    dry_run: bool
    triggered_by: str                 # "scheduler" | "manual" | "wizard"
    tz: str                           # for midnight-reset math
```

Phases 3–6 add nothing to this contract except reading `ctx.dry_run` before any outbound side effect (email, form submit).

### Pattern 5: Rate-limit envelope (pre-run gate)

**What:** `RateLimiter` holds:
- `daily_cap` (default 20)
- `delay_min_s` / `delay_max_s` (30 / 120)
- Current day's counter in the `runs` table aggregated at read time, or (simpler) a `rate_limit_counters(date, submitted_count)` table with a single row per local-TZ date.

**How it gates:**
- `await_precheck()` called by `SchedulerService.run_pipeline()` before creating the `Run` row. Checks `submitted_count < daily_cap`. If over, raises `RateLimitExceeded` which the scheduler catches and logs `run_skipped(reason="daily_cap_hit")`.
- `midnight_reset()` runs as an APScheduler job at `CronTrigger(hour=0, minute=0, timezone=settings.tz)` and inserts the next day's counter row.
- `random_action_delay()` returns `random.uniform(delay_min_s, delay_max_s)` — stored here in Phase 1 but only called by Phase 6 submitters.

**Anti-pattern avoided:** No token bucket, no in-memory counter. The DB row is the source of truth so the dashboard reads the same number the gate enforces.

```python
# app/scheduler/rate_limit.py
import random
from datetime import date
from zoneinfo import ZoneInfo

class RateLimitExceeded(Exception): pass

class RateLimiter:
    def __init__(self, daily_cap, delay_min, delay_max, tz):
        self.daily_cap = daily_cap
        self.delay_min = delay_min
        self.delay_max = delay_max
        self.tz = ZoneInfo(tz)

    def _today_local(self) -> date:
        from datetime import datetime
        return datetime.now(self.tz).date()

    async def await_precheck(self):
        count = await get_submitted_count_for(self._today_local())
        if count >= self.daily_cap:
            raise RateLimitExceeded(f"daily cap {self.daily_cap} reached")

    def random_action_delay(self) -> float:
        return random.uniform(self.delay_min, self.delay_max)

    async def record_submission(self):
        await increment_counter_for(self._today_local())
```

### Pattern 6: Settings row as single source of truth

All toggles live in one `settings` row (single-row table, `id=1`). No in-memory globals, no env-var overrides at runtime. This is the "UI as view over DB" pattern — the dashboard reads `settings`, the scheduler reads `settings`, the API writes `settings`. On startup, `KillSwitch` and `SchedulerService` hydrate themselves from this row.

### Anti-patterns to avoid (Phase 1 specific)

- **Two-process split (web + worker):** Keep everything in one FastAPI process. Anti-pattern for single-user, adds shared-state pain.
- **Storing secrets anywhere other than the encrypted `secrets` table:** no env seeding, no config files.
- **Logging the `RunContext` or `Settings` objects verbatim:** always use `ctx.redacted_dict()` or explicit fields.
- **`if settings.dry_run:` scattered across handlers:** Phase 1 centralizes the check inside `RunContext` and at submission boundaries (the only places Phase 3+ need to branch).
- **APScheduler 4.0 alpha:** do not ship on alphas.
- **`--workers 2`:** breaks singleton scheduler state silently.

## SQLModel Schema (Phase 1 tables)

Minimal schema the planner should implement. Designed so Phases 2–7 **add columns / tables** without renaming or migrating Phase 1 rows.

```python
# app/db/models.py
from datetime import datetime
from typing import Optional
from sqlmodel import SQLModel, Field
from sqlalchemy import Column, LargeBinary, Text, JSON

class Settings(SQLModel, table=True):
    __tablename__ = "settings"
    id: int = Field(default=1, primary_key=True)           # enforced single-row

    # Safety toggles
    kill_switch: bool = Field(default=False)
    dry_run: bool = Field(default=False)

    # Rate envelope
    daily_cap: int = Field(default=20)
    delay_min_seconds: int = Field(default=30)
    delay_max_seconds: int = Field(default=120)
    timezone: str = Field(default="UTC")                   # cached from TZ env at wizard time

    # Wizard state
    wizard_complete: bool = Field(default=False)

    # Meta
    updated_at: datetime = Field(default_factory=datetime.utcnow)

class Secret(SQLModel, table=True):
    __tablename__ = "secrets"
    id: int = Field(default=None, primary_key=True)
    name: str = Field(index=True, unique=True)             # e.g. "anthropic_api_key", "smtp_password"
    ciphertext: bytes = Field(sa_column=Column(LargeBinary, nullable=False))
    created_at: datetime = Field(default_factory=datetime.utcnow)
    updated_at: datetime = Field(default_factory=datetime.utcnow)

class Run(SQLModel, table=True):
    __tablename__ = "runs"
    id: int = Field(default=None, primary_key=True)

    # Lifecycle
    started_at: datetime = Field(default_factory=datetime.utcnow, index=True)
    ended_at: Optional[datetime] = None
    duration_ms: Optional[int] = None
    status: str = Field(default="running")                 # running | succeeded | failed | skipped
    failure_reason: Optional[str] = None                   # None | killed | rate_limit | error | dry_run_skip | ...

    # Context at run start
    dry_run: bool = Field(default=False)
    triggered_by: str = Field(default="scheduler")         # scheduler | manual | wizard

    # Per-stage counts — zero in Phase 1 but present so Phase 2+ only populates
    counts: dict = Field(default_factory=dict, sa_column=Column(JSON))
    # Shape: {"discovered": 0, "matched": 0, "tailored": 0, "submitted": 0, "failed": 0}

class RateLimitCounter(SQLModel, table=True):
    __tablename__ = "rate_limit_counters"
    day: str = Field(primary_key=True)                     # ISO date in local TZ
    submitted_count: int = Field(default=0)
```

**Schema notes:**
- `counts` is a JSON blob, not separate columns, so Phases 2+ can add stage names without migrations.
- `failure_reason` is a free string rather than an enum so Phase 1 does not have to predict every future reason. Document the canonical set in a module-level constant.
- `Settings` is a one-row table (convention: always `id=1`). Simpler than key-value — every flag is a column and thus type-safe.
- `Run.status='running'` serves as the DB sentinel for the run-lock; a cleanup on startup marks any leftover `running` rows as `failed` / `reason='crashed'`.
- Indexes: `runs(started_at)` desc for the "last 50" query; `secrets(name)` unique.

## Fernet Secrets Pattern

### Fail-fast key loading

```python
# app/security/fernet.py
from cryptography.fernet import Fernet, InvalidToken
import base64

class InvalidFernetKey(Exception): pass

class FernetVault:
    def __init__(self, fernet: Fernet):
        self._fernet = fernet
        self._known_plaintexts: set[bytes] = set()        # fed to log scrubber

    @classmethod
    def from_env(cls, key_str: str) -> "FernetVault":
        if not key_str:
            raise InvalidFernetKey("FERNET_KEY env var is required")
        try:
            # Fernet validates length + base64 shape here
            fernet = Fernet(key_str.encode() if isinstance(key_str, str) else key_str)
        except (ValueError, TypeError) as e:
            raise InvalidFernetKey(f"FERNET_KEY is not a valid Fernet key: {e}") from e
        return cls(fernet)

    def encrypt(self, plaintext: str) -> bytes:
        return self._fernet.encrypt(plaintext.encode("utf-8"))

    def decrypt(self, ciphertext: bytes) -> str:
        try:
            pt = self._fernet.decrypt(ciphertext).decode("utf-8")
        except InvalidToken as e:
            raise InvalidFernetKey("stored secret cannot be decrypted — key may have changed") from e
        self._known_plaintexts.add(pt.encode("utf-8"))
        return pt
```

**Key contract:**
- App fails to start if `FERNET_KEY` is missing or malformed — lifespan raises before scheduler binds port. Docker `restart: unless-stopped` will give the user a tight loop they can see in `docker compose logs`.
- If a stored secret can't be decrypted (user changed the key), the UI surfaces "secret is unreadable, re-enter it". Do not auto-delete — preserve for forensic.
- `Fernet.generate_key()` is the canonical way users produce a key. `.env.example` documents `FERNET_KEY=$(python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())")`.

### Avoiding plaintext leakage through SQLAlchemy echo

- Set `create_async_engine(..., echo=False)` unconditionally in prod. If `echo=True` is used in dev, the `LargeBinary` ciphertext column will be repr'd but the plaintext never will — but don't rely on that, keep echo off.
- Secrets table never has a plaintext column. `SecretCreate` Pydantic input model takes `SecretStr` and is converted to `ciphertext` inside the service layer before any DB write.
- **Every time a secret is decrypted, register its plaintext with the log scrubber** via `FernetVault.decrypt()` (see above). This way the scrubber knows the value of, say, the Anthropic API key even though it was never hardcoded.

Sources: [Fernet docs](https://cryptography.io/en/latest/fernet/), [pyca/cryptography#6771](https://github.com/pyca/cryptography/issues/6771) (confirms InvalidToken semantics).

## Log Scrubbing (zero-tolerance PII/secret leakage)

### Strategy

Two layers of defense, both mandatory:

1. **`logging.Filter` on the root logger** — catches every record from every logger (stdlib, uvicorn, SQLAlchemy, APScheduler, third parties).
2. **`structlog` processor** mirroring the same pattern set — catches anything emitted via structlog before it reaches the stdlib handlers.

Both pull from a single `SecretRegistry` singleton so adding a secret at runtime (e.g. user enters Anthropic key in Settings page) immediately propagates to both paths.

### Pattern registry

```python
# app/security/log_scrubber.py
import logging
import re
from threading import Lock

class SecretRegistry:
    """Single source of truth for strings that must never appear in logs."""
    def __init__(self):
        self._literals: set[str] = set()
        self._patterns: list[re.Pattern] = []
        self._lock = Lock()
        # Pre-register static patterns for common leak shapes
        self._patterns.append(re.compile(r"sk-ant-[A-Za-z0-9\-_]{20,}"))      # Anthropic
        self._patterns.append(re.compile(r"sk-[A-Za-z0-9]{32,}"))              # OpenAI style
        self._patterns.append(re.compile(r"gAAAAA[A-Za-z0-9\-_=]{20,}"))      # Fernet token shape
        self._patterns.append(re.compile(r"(?i)password[\"'=:\s]+[^\s\"']{4,}"))

    def add_literal(self, value: str) -> None:
        with self._lock:
            if value and len(value) >= 4:
                self._literals.add(value)

    def scrub(self, text: str) -> str:
        if not isinstance(text, str):
            return text
        with self._lock:
            for lit in self._literals:
                if lit in text:
                    text = text.replace(lit, "***REDACTED***")
            for pat in self._patterns:
                text = pat.sub("***REDACTED***", text)
        return text

REGISTRY = SecretRegistry()

class RedactingFilter(logging.Filter):
    def filter(self, record: logging.LogRecord) -> bool:
        # Scrub the pre-formatted message
        if isinstance(record.msg, str):
            record.msg = REGISTRY.scrub(record.msg)
        # Scrub args (for %-style formatting)
        if record.args:
            record.args = tuple(
                REGISTRY.scrub(a) if isinstance(a, str) else a for a in record.args
            )
        return True

def structlog_scrub_processor(_logger, _name, event_dict):
    for k, v in list(event_dict.items()):
        if isinstance(v, str):
            event_dict[k] = REGISTRY.scrub(v)
    return event_dict
```

### Wiring

```python
# app/logging_setup.py
import logging
import sys
import structlog
from app.security.log_scrubber import RedactingFilter, structlog_scrub_processor

def configure_logging(level: str, log_dir):
    log_dir.mkdir(parents=True, exist_ok=True)

    redacting_filter = RedactingFilter()

    # Stdlib handlers
    stdout = logging.StreamHandler(sys.stdout)
    stdout.addFilter(redacting_filter)
    file = logging.FileHandler(log_dir / "app.log", encoding="utf-8")
    file.addFilter(redacting_filter)

    root = logging.getLogger()
    root.handlers = [stdout, file]
    root.setLevel(level)

    # Filter also on EVERY logger via propagation — attaching to root is sufficient
    # because filters on handlers run after propagation.

    structlog.configure(
        processors=[
            structlog.contextvars.merge_contextvars,
            structlog.processors.add_log_level,
            structlog.processors.TimeStamper(fmt="iso"),
            structlog_scrub_processor,                     # redact BEFORE rendering
            structlog.processors.JSONRenderer(),
        ],
        wrapper_class=structlog.make_filtering_bound_logger(logging.INFO),
        logger_factory=structlog.stdlib.LoggerFactory(),
        cache_logger_on_first_use=True,
    )
```

### Runtime secret registration

Every place that handles a known secret calls `REGISTRY.add_literal(value)`:

- On app start: `FernetVault.from_env` registers the master key (Fernet keys should never appear, but belt + braces).
- When the UI writes a secret (`POST /settings/secrets`): the service layer calls `REGISTRY.add_literal(plaintext)` before encrypting and storing.
- On startup: after loading `secrets` table, decrypt each → register → discard plaintext. This is why `FernetVault.decrypt()` auto-adds to the registry.

## Kill-switch Implementation (recap of Pattern 3)

See Pattern 3 above for code. Key implementation notes for the planner:

1. **Event-based signal** — `asyncio.Event` is awaitable; stages can `await killswitch._event.wait()` if they want to short-circuit.
2. **Task cancellation** — `SchedulerService._current_task.cancel()` is the hard-stop. Propagates `CancelledError` through every `await`.
3. **APScheduler-level pause** — `scheduler.pause_job("hourly_heartbeat")` stops future ticks. `resume_job` re-enables.
4. **DB settings write** — the `settings.kill_switch` column is written so the state persists across restarts. Lifespan reads it on boot and engages the killswitch (but does not cancel anything, since no run is in flight yet).
5. **UI trigger** — `POST /toggles/kill-switch` with HTMX, returns the updated status partial. One click.
6. **Post-mortem** — any `Run` row left in `status='running'` when cancellation handler fires gets `status='failed'`, `failure_reason='killed'`. In Phases 3+ this extends to marking partial per-stage records as well.

## Dry-run Plumbing

- **Storage:** `settings.dry_run: bool` (single row).
- **Read point:** `SchedulerService.run_pipeline()` reads it when constructing the `RunContext`.
- **Stamping:** `Run.dry_run` is set at creation and never changes mid-run (so toggling the switch during a run does not retroactively relabel records).
- **Propagation:** `RunContext.dry_run` is passed to every stage. In Phase 1 there are no downstream stages — but the stub pipeline function still receives the context so the contract exists.
- **Stage behavior contract (documented in Phase 1, enforced in Phases 3+):** When `ctx.dry_run is True`, stages may perform reads and may write observability rows (with `dry_run=True`), but must not call any submitter, email sender, or external-effect method. Submitters and notifier classes raise `RuntimeError` if invoked with `ctx.dry_run=True` as a defensive final check.
- **UI affordance:** dashboard pill says "Dry-run: ON" in yellow when enabled. Run history rows with `dry_run=True` are rendered with a yellow badge.

The dashboard toggle writes to `settings.dry_run` and the next scheduled run picks it up. There's no mid-run toggle — by design.

## Rate Limiter (recap)

See Pattern 5 above for code. Planner notes:

- **Pre-check is at `SchedulerService.run_pipeline()` entry.** Raises `RateLimitExceeded`. Scheduler catches, logs a `run_skipped` event with `failure_reason='rate_limit'`, writes a `Run` row with status `skipped` for visibility on the dashboard.
- **Midnight reset via APScheduler job**, not cron-inside-code. Handles DST properly because `CronTrigger(timezone=ZoneInfo(tz))` respects wall-clock time.
- **Counter is a dedicated `rate_limit_counters(day, submitted_count)` table** keyed by local-TZ ISO date. One row per day. Simpler than aggregating `runs` on the fly and easier to display on the dashboard.
- **Phase 1 never increments the counter** (no submissions happen). The increment call site is `RateLimiter.record_submission()`, called by Phase 6's submitter. Phase 1 includes a unit test that calls it directly and asserts counter math.
- **Randomized per-action delay** lives on the same `RateLimiter` but is only consumed in Phase 6. Phase 1 exposes the configured range on the Settings page and validates it (min < max, both > 0, max < 600).

## First-Run Setup Wizard

### State model

- Single boolean `settings.wizard_complete`.
- A FastAPI dependency `require_wizard_or_redirect` is applied to `GET /`; if `wizard_complete=False`, it returns `RedirectResponse("/setup/1")`.
- The wizard itself never redirects to `/` until either (a) the user completes step 3, (b) the user clicks "Skip setup". Both paths set `wizard_complete=True`.
- Skip is a one-click `POST /setup/skip` that confirms "You can finish these later in Settings."

### Steps

1. **`/setup/1` Upload resume** — form with `<input type="file">`, server stores DOCX at `data/uploads/resume_base.docx`. Does not parse yet (Phase 4 does that). On success → `/setup/2`.
2. **`/setup/2` API keys & SMTP** — form fields for `anthropic_api_key`, `smtp_host`, `smtp_port`, `smtp_user`, `smtp_password`. All `SecretStr`. On save → encrypt via `FernetVault`, write to `secrets` table, register with log scrubber. → `/setup/3`.

## Phase 1-Specific Pitfalls (from PITFALLS.md + research)

### From PITFALLS.md Pitfall 5 — Sensitive Data Leakage

The log scrubber and PII boundary are architectural, not decorative. Phase 1 must ship the assertion test. Phase 2+ inherits the registry but adds its own redaction contracts (e.g. resume content never enters LLM prompts unredacted — Phase 3 concern).

### APScheduler singleton lock under gunicorn / multiple workers

**Trap:** Running with `uvicorn --workers 2` or gunicorn will spawn two schedulers that both fire on the hour, doubling runs and corrupting counters.
**Prevention:** `--workers 1` is enforced in `Dockerfile` CMD AND documented in a comment AND the `/health` endpoint can include `worker_pid` for sanity-checking in logs.

### Fernet key rotation trap

**Trap:** User rotates `FERNET_KEY` between restarts; existing ciphertext becomes garbage. Silent data loss unless flagged loudly.
**Prevention:** On startup, try to decrypt one known secret; if it fails with `InvalidToken`, surface a big red banner on the dashboard: "Stored secrets cannot be decrypted. The master key changed. Re-enter your API keys in Settings." Do NOT delete the rows — keep them so the user knows what to re-enter. Document rotation = loss.

### DB sentinel run-lock orphan rows

**Trap:** Container killed mid-run leaves `Run(status='running')` in DB. On next boot the UI shows a stuck "running" row forever, and the run-lock logic might be confused.
**Prevention:** `SchedulerService.start()` runs a one-shot `mark_orphans_failed` query at boot: `UPDATE runs SET status='failed', failure_reason='crashed' WHERE status='running'`.

### APScheduler jobstore migration pain

**Trap:** Schema of `SQLAlchemyJobStore` tables is managed by APScheduler itself, NOT by Alembic. Mixing them in the same DB file is fine, but do not put APScheduler tables under Alembic's `target_metadata`.
**Prevention:** In `alembic env.py`, exclude the `apscheduler_jobs` table via `include_object` filter. Document this.

### SQLite WAL mode

**Trap:** Default `journal_mode=DELETE` causes writers to block readers. Dashboard HTMX polling + scheduler writing the `runs` table can cause lock contention.
**Prevention:** On first connect, run `PRAGMA journal_mode=WAL`. Do this once in `init_db()`. WAL tolerates concurrent reader/writer.

### TZ env var not applied to APScheduler

**Trap:** Setting `TZ=America/Los_Angeles` in compose.yml updates the OS timezone but APScheduler defaults to UTC unless explicitly told.
**Prevention:** Read `TZ` via pydantic-settings, pass `ZoneInfo(settings.tz)` to every `AsyncIOScheduler(timezone=...)` and `CronTrigger(timezone=...)` constructor. Unit test that next-run time is local-midnight when configured so.

### Structlog vs stdlib redaction ordering

**Trap:** If the structlog processor chain renders to string before scrubbing, the redaction pass operates on the final JSON string rather than typed values — patterns may match but also over-match (redacting keys, etc.).
**Prevention:** Place `structlog_scrub_processor` BEFORE `JSONRenderer` in the chain so it operates on the event dict, not the rendered string. Test asserts both paths.

### HTMX polling DoS

**Trap:** Aggressive polling intervals (1s) from many open tabs could hammer the DB.
**Prevention:** 5s for status pill, 15s for countdown. Add `hx-trigger="every 5s[document.hidden === false]"` to pause polling when tab hidden.

### Writing settings mid-run

**Trap:** User toggles `dry_run` while a run is in progress; a stage that rereads settings mid-flight could see inconsistent state between records from the same run.
**Prevention:** `RunContext` snapshots settings at run-start and never rereads them. Stages read from `ctx`, not from the DB.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---|---|---|---|
| Symmetric encryption of API keys | Custom AES + HMAC | `cryptography.fernet.Fernet` | Battle-tested, gives AEAD + versioning for free. |
| Hourly job with overlap protection | `asyncio.create_task` + manual sleeps | `APScheduler 3.11 AsyncIOScheduler` + `max_instances=1` | Handles coalesce, misfire, timezone, persistence out of the box. |
| Structured logging | `json.dumps(dict)` + print | `structlog` with processor chain | Processors are the right seam for redaction; JSON rendering is one line. |
| Log secret redaction | `message.replace(api_key, "***")` ad hoc at call sites | Root-level `logging.Filter` + structlog processor pulling from `SecretRegistry` | Central, auditable, testable. A single miss is a breach. |
| SQLite session lock | `fcntl.flock` on a pidfile | `asyncio.Lock` + `Run(status='running')` sentinel under `--workers 1` | File locks add cross-platform pain; we own the process. |
| Task cancellation | Boolean flag polled in a loop | `asyncio.Task.cancel()` + `CancelledError` propagation | Native, works through every await. |
| Toggle state persistence | JSON file + reload | `settings` SQLite row | Survives restart, visible in DB browser, one source of truth. |
| CSS framework | Hand-written CSS | `Pico.css` served statically | Semantic, tiny, no build. |
| Wizard guard | Custom middleware | FastAPI dependency returning `RedirectResponse` | Idiomatic, testable with TestClient. |
| `.env` parsing | `os.environ.get` scattered | `pydantic-settings` BaseSettings class | Typed, validated, fails fast. |

**Key insight:** Phase 1 is 100% plumbing. Every decision should err toward "use the library's answer, not the clever one". Custom code here multiplies maintenance for every downstream phase.

## Code Examples

### Full `config.py` with fail-fast

```python
# app/config.py
from pathlib import Path
from pydantic import Field, field_validator
from pydantic_settings import BaseSettings, SettingsConfigDict
from cryptography.fernet import Fernet

class Settings(BaseSettings):
    model_config = SettingsConfigDict(env_file=".env", extra="ignore")

    fernet_key: str = Field(..., alias="FERNET_KEY")
    tz: str = Field("UTC", alias="TZ")
    bind_address: str = Field("0.0.0.0", alias="BIND_ADDRESS")
    data_dir: Path = Field(Path("/data"))
    log_level: str = Field("INFO")

    @field_validator("fernet_key")
    @classmethod
    def _validate_fernet(cls, v: str) -> str:
        if not v:
            raise ValueError("FERNET_KEY is required")
        try:
            Fernet(v.encode())
        except Exception as e:
            raise ValueError(f"FERNET_KEY is not a valid Fernet key: {e}") from e
        return v

settings = Settings()
```

## Open Questions

1. **Should wizard completion be persistent even if the user deletes `data/uploads/resume_base.docx`?**
   - What we know: completion is a DB flag, resume file is on disk. They can drift.
   - Recommendation: On wizard complete, store `resume_uploaded_at` in settings. Dashboard shows a warning banner if the file path is missing but the flag is set. Phase 4 handles re-upload UX.

2. **Where does `keywords` live — in `settings` row or its own table?**
   - Recommendation: For Phase 1, a `keywords_csv: str` column on `settings`. Phase 2 promotes it to a `keywords` table if per-keyword metadata (weight, last_hit) is needed.

3. **Timezone change mid-deployment.**
   - If user changes `TZ` env var and restarts, the midnight-reset moves. Counters from the previous day may or may not be correct. Recommendation: document as "don't change TZ unless you mean it", and include TZ in the `rate_limit_counters` row if you want to be defensive (probably overkill for Phase 1).

4. **Should `/health` require authentication?**
   - Phase 1 is single-user LAN-bound. No auth in v1 per CONTEXT.md (multi-user is deferred). `/health` is unauthenticated and returns minimal info.
