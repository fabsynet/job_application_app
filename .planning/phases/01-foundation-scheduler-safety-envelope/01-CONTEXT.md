# Phase 1: Foundation, Scheduler & Safety Envelope - Context

**Gathered:** 2026-04-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Bootable Docker container with encrypted secrets, hourly scheduler with run-lock, kill-switch, dry-run toggle, rate-limit envelope, and PII-safe structured logging. No actual job discovery, tailoring, or submission happens in this phase — this is the safety frame that every later phase runs inside.

Requirements in scope: FOUND-01..07, DISC-07, SAFE-01, SAFE-02, SAFE-03.

</domain>

<decisions>
## Implementation Decisions

### Secrets Bootstrap
- **Master key delivery**: `FERNET_KEY` env var passed at `docker compose up` time. Not generated by the app, not mounted as a file. User stores the key outside the container.
- **API key / SMTP entry**: Web UI form (Settings page). Fields are encrypted with the master key and stored in the SQLite secrets table on save. No env-var seeding — UI is the only entry path.
- **Key rotation / loss**: No rotation tool in v1. If the master key is lost or changed, stored secrets become unreadable; user re-enters API keys and SMTP credentials via the UI. Document this explicitly so it is not a surprise.
- **Leak prevention**: Zero tolerance. Implement a global log scrubber that redacts any value matching registered secret patterns (API keys, passwords, Fernet key, SMTP creds) before it reaches stdout, log files, or error responses. Ship with a test case that asserts known-secret patterns never appear in log output.

### Control Panel UX (Phase 1 landing surface)
- **Home page is a live status dashboard** showing: scheduler state (running / paused / killed), next scheduled run time, last run timestamp + duration + outcome, counts from the last run (discovered / matched / tailored / submitted / failed — all zero in Phase 1), and prominent toggles for **Kill switch** and **Dry run**.
- **Kill-switch behavior**: Hard stop. Cancels the in-flight run immediately; any partially-processed records in that run are marked `failed` with reason `killed`. Scheduler pauses future runs until the switch is flipped back.
- **Dry-run scope**: Real reads, no writes. When dry-run is ON, scrapers actually fetch, LLM actually runs (in later phases), and database rows for observability are still written — but **no outbound submissions** (email send, browser form submit) and **no outbound notification emails**. A dry-run flag is stamped on every record created during the run so it is easy to tell simulated state from real state.
- **Run history**: DB keeps every run record forever (small footprint — hourly runs produce ~720 rows/month, trivial for SQLite). UI shows the 50 most recent runs by default with a "show more" affordance for older rows. No pruning job.

### Rate-Limit Envelope Defaults
- **Daily application cap**: 20/day by default. User-configurable in Settings.
- **Per-action delay within a browser session**: randomized 30–120 seconds. Phase 1 defines and stores this range; Phase 6 is the first phase that actually uses it.
- **Cap scope**: Single global cap across all sources. No per-source override in v1.
- **Counter reset**: Local midnight in the container's timezone (driven by the `TZ` env var). Not a 24-hour rolling window.
- The rate-limit envelope is enforced by the scheduler before any downstream stage exists, so later phases inherit throttling for free.

### Docker & Volumes
- **Deploy shape**: `docker compose up`. Ship a `compose.yml` in the repo with the web service, volume mounts, and `.env` file references. A `.env.example` documents required variables (`FERNET_KEY`, `TZ`, `BIND_ADDRESS`, etc.).
- **Host data layout**: Single `./data` volume mounted into the container. Everything lives under it — SQLite db file, logs, uploaded base DOCX resumes, Playwright storageState (for Phase 6), tailored resume artifacts (for Phase 4+). One directory to back up, one directory to wipe for a reset.
- **Network binding**: LAN-accessible by default (`0.0.0.0`) so the user can hit the UI from another device on their home network. The bind address is configurable via a `BIND_ADDRESS` env var in case the user wants to restrict it to `127.0.0.1`.
- **First-run experience**: Setup wizard. When the container boots with an empty `./data` volume, the UI routes to a wizard that walks the user through (1) upload base DOCX resume, (2) enter Claude API key + SMTP credentials, (3) set target keywords. The wizard writes the wizard-complete flag to the DB when done; subsequent boots go straight to the dashboard. In Phase 1 the wizard lands the user on the dashboard even if later-phase inputs (keywords, API keys) are unused yet — the shell exists from day one so Phase 2 only fills in the fields, not the scaffolding.

### Claude's Discretion
- Exact schema of the `runs` and `secrets` tables (as long as the log-scrubber test and the 50-row UI work).
- Exact UI framework shape (HTMX + Jinja templates are locked by STACK.md; the component/page breakdown within Phase 1 is Claude's call).
- How the setup-wizard steps are ordered and rendered, as long as all three steps are reachable and completion is persistent across restarts.
- Exact format of structured log entries (JSON shape, field names) as long as PII scrubbing is enforced.
- APScheduler version choice (3.x stable vs 4.x async-native) — STACK.md flags this; planner/researcher can pick.

</decisions>

<specifics>
## Specific Ideas

- The kill-switch and dry-run toggles must be **one click away from the home page** — not buried in Settings. They are safety primitives and should feel like big red buttons.
- The status dashboard should make it obvious at a glance whether the scheduler is healthy — one colored dot + one-line status ("Running, next run in 47 min" / "Paused by kill-switch" / "Dry-run mode").
- The setup wizard must not block the user from reaching the dashboard if they want to skip — wizard is guidance, not a gate.
- "Zero PII in logs" is a testable property, not a vibe. The planner should write an explicit assertion for it.

</specifics>

<deferred>
## Deferred Ideas

- **Key rotation tool** (CLI to re-encrypt stored secrets with a new Fernet key) — not in v1; user re-enters secrets if the key is lost.
- **Per-source rate limit overrides** (e.g. stricter cap for LinkedIn) — v1.x when LinkedIn lands in the LINK-* requirements. Phase 1 sets the envelope as global-only.
- **Log pruning / retention policy** — not needed at current scale; revisit only if DB growth becomes a problem.
- **Observability export** (Prometheus/OpenTelemetry metrics) — nice-to-have; v2+.
- **Multi-user / role-based access on the LAN-bound UI** — out of scope; single-user per deployment is a project-level decision.

</deferred>
